hello-spec · explained without the jargon

Teaching computers to check code safely

A small, working demonstration that brings together two open security standards — and shows how to make an AI prove its work.

🏠 Think of it like a building inspection. One standard is the building code — the rulebook of what's safe. The other is how a trustworthy inspector works — never guessing, always showing evidence, getting a second opinion, and leaving the owner in charge. hello-spec is a model house we walk through to see it all in action.
The big picture

Three pieces fit together

Project CodeGuard

The building code

A shared rulebook of safe-coding practices — "don't do it this risky way, do it this safe way." It's knowledge, written down.

Foundry Security Spec

The trustworthy inspector

A blueprint for an AI that hunts for security problems — and only reports the ones it can prove, so you can trust the result.

hello-spec

The model house

A tiny working demo that runs both: it inspects a small program built with deliberate flaws — and a safe twin with none.

How it lines up

What we took from each standard

This is the heart of the project: it doesn't reinvent anything — it combines two existing standards. The rulebook comes from one; the trustworthy-inspector design comes from the other.

CodeGuard the rules

  • Safe-coding rules written in CodeGuard's own format
  • Checked & packaged by CodeGuard's real tools
  • The fixes apply CodeGuard's recommended safe practices

Foundry the inspector

  • A team of specialist inspectors, each with one job
  • The evidence rule: a problem is only "real" once proven
  • An independent second inspector confirms a genuine break
  • A written "constitution" of 11 trust rules it must obey
  • It proposes fixes, but never changes anything itself

Go deeper · the core idea

The life of a finding

Why can you trust what the inspector says? Because every problem has to survive these six steps before a human ever sees it.

1

Spot

An inspector notices something that looks risky.

2

Prove

It must point to the exact evidence in the code.

3

Confirm

A different inspector independently re-checks it.

4

Write up

Only proven problems get written down.

5

Propose

A suggested fix is drafted — and tested.

6

You decide

Nothing changes unless a person says so.

Three more things worth knowing

🛡️ What keeps it honest trust ▸

The inspector follows a written "constitution." A few of its rules, in plain language:

No claim without proof. A confident-sounding answer isn't enough — the evidence has to check out.
Only a fresh checker confirms a break. The one who found it can't also be the one who "verifies" it.
The human outranks the machine. People can overrule any automated decision, and it's recorded.
Nothing changes without permission. Fixes are proposed for review — never applied behind your back.
📐 How it was built process ▸

It wasn't just coded ad-hoc. It followed a disciplined, write-it-down-first process — and the newest part (the fix-proposer) was built entirely this way:

Agree the rules→ Write the spec→ Plan it→ Break into tasks→ Build→ Check against the constitution

Every step is checked against the project's own "constitution," so the build can't quietly drift away from its own principles.

🤖 A real AI, kept on a leash why it matters ▸
🤖🔗

A genuine AI model does the clever judging here — but it's only believed when it can point to proof. That's the whole idea: harness the cleverness, don't trust it blindly. The same machinery works with the AI switched off (using fixed rules), so its behaviour can always be checked.

Is this real? The machinery is real and runs end-to-end. The program it inspects is a tiny teaching example with deliberate flaws (and a safe twin) — not a real product. The point is to show the pattern: rules you can trust, an inspector that proves its work, and a human always in charge.
